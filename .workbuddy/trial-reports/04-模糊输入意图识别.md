## 测试报告 — 模糊输入意图识别

**测试日期**：2026-04-22
**测试角色**：模糊输入-D（模拟不擅长表达的普通用户）
**测试方法**：通过阅读 SKILL.md 规则，模拟每个场景的预期行为

---

### 场景1：「这事你怎么看？」

- **预期行为**：按 SKILL.md 第54-57行规则，这是**完全模糊的输入**，没有任何主题。Skill 应触发追问机制，向用户确认：
  1. 是在面临具体决策，还是纯粹学习？
  2. 如果是决策，决策场景是什么？
  3. 有没有明确想了解的事件？

- **实际评估**：规则覆盖了这个场景，但追问设计存在冗余。「这事你怎么看？」连"什么事"都没说，用户最可能的第一步回答是「就是XX事件啊」。追问的前两个问题（决策还是学习、决策场景）在用户澄清主题前意义不大。

- **结果**：部分覆盖

- **问题/建议**：
  - 问题：追问顺序不够优化。"有没有明确想了解的事件？"应该是第一个问题，因为其他问题都依赖于此。
  - 建议：在追问流程中增加**前置问题**：「你说的'这事'是指什么？」，获得主题后再进入标准追问流程（决策还是学习→决策场景）。

---

### 场景2：「给我说说诺基亚的事」

- **预期行为**：用户提到了具体事件（诺基亚），但没说想要什么。按规则，Skill 可能识别为"经验学习"维度（description 中有「给我说说XX」触发词），然后尝试蒸馏诺基亚。但用户没说清是想学习了解还是面临类似决策。

- **实际评估**：
  - 正面：Skill 会识别出事件主题（诺基亚）
  - 负面：Skill 可能会跳过追问，直接进入蒸馏流程（因为有具体事件名）
  - 问题：规则说「如果用户输入模糊，通过追问明确」，但「给我说说XX」在 description 里被列为触发词而非模糊输入，这两处存在矛盾

- **结果**：未通过

- **问题/建议**：
  - 问题：description 第13行「给我说说XX」被列为触发词，但 SKILL.md 正文第54行说"如果用户输入模糊，通过追问明确"——两处规则不一致。测试员模拟 Skill 执行时，无法判断应该追问还是直接蒸馏。
  - 建议：明确「给我说说XX」是否算"模糊输入"。如果用户只是说「给我说说XX」而没说想从中学什么，应该追问。规则中增加：「用户提到具体事件但未说明意图 → 追问：想学习决策逻辑？还是有类似决策困境想参考？」

### 场景3：「我最近在想要不要转型，做了3年现在遇到瓶颈了，但转型风险也大，有没有什么类似的案例可以参考？」

- **预期行为**：这是**反向蒸馏**的典型场景。SKILL.md 第128-141行详细描述了反向蒸馏：
  1. 提取用户处境的关键特征（行业、阶段、资源、困境）
  2. 追问行业/规模等关键信息来提高匹配精度（第134行）
  3. 降级处理：信息不足时告知用户

- **实际评估**：
  - 用户已提供了较多信息：转型困境、3年经验、瓶颈感、风险顾虑
  - Skill 应该识别为反向蒸馏，并追问：什么行业？什么规模的组织？转型的方向是什么（跨行业？转赛道？）
  - 降级条件：第141行写了「如果搜索匹配到的案例少于3个，或相关度低于阈值，告知用户」

- **结果**：通过

- **问题/建议**：
  - 优点：反向蒸馏规则完整，匹配逻辑清晰
  - 细节建议：用户说"有没有类似的案例可以参考"已经暗示了反向蒸馏意图，Skill 应该优先识别这句话，而不是从其他维度（如"战略选择"）切入

### 场景4：「帮我分析一下OpenAI跟马斯克那事，顺便看看现在AI行业是不是跟当年互联网泡沫一样」

- **预期行为**：这是**混合意图**，包含两个独立需求：
  1. 单事件蒸馏：OpenAI与马斯克的纠纷（单事件）
  2. 多事件蒸馏：AI行业 vs 互联网泡沫（需要跨时间维度的对比分析）

- **实际评估**：
  - SKILL.md 的多事件蒸馏规则（第111-127行）主要描述的是"用户提出决策场景但未指定具体事件"的情况
  - 场景4的「顺便看看现在AI行业是不是跟当年互联网泡沫一样」是明确的对比请求，但没有明确的决策场景描述
  - Skill 可能只处理第一个需求（OpenAI），把第二个需求当成"多事件蒸馏"但不知道决策场景是什么
  - 规则没有明确说明如何处理**多个独立意图**的输入

- **结果**：未通过

- **问题/建议**：
  - 问题：SKILL.md 没有处理"混合意图"（多个独立蒸馏需求）的规则。当用户说「帮我分析XX，顺便看看YY」时，Skill 没有判断逻辑来决定：是同时处理？还是确认优先级？还是询问是否作为同一决策场景的两个维度？
  - 建议：增加"意图拆分"规则：
    - 如果用户明确提到两个不相关事件 → 分别处理，或询问用户优先级
    - 如果第二个事件有决策场景隐含 → 作为多事件蒸馏处理
  - 场景4中，「AI行业 vs 互联网泡沫」本身就是一种多事件蒸馏请求，Skill 应该能识别并询问：「你是想了解AI行业会不会重蹈互联网泡沫的覆辙吗？有没有具体的担忧或决策问题？」

## 综合评估

### 覆盖度分析

| 测试维度 | 评分 | 说明 |
|---------|------|------|
| **意图识别规则的覆盖度** | 中 | 6个决策维度覆盖较全，但"模糊输入"的定义不清晰（「给我说说XX」算不算模糊？） |
| **追问逻辑的完整性** | 低-中 | 追问3问有用，但顺序不优化（场景1）、与触发词列表矛盾（场景2）、无多意图拆分规则（场景4） |
| **不触发场景的拦截能力** | 高 | 规则明确，执行路径清晰 |
| **蒸馏模式判断准确性** | 中 | 单事件/多事件/反向蒸馏的区分逻辑基本清晰，但边界有模糊地带（场景4） |

### 发现的问题

1. **模糊输入的定义不一致**
   - SKILL.md 第54行说"如果用户输入模糊，通过追问明确"
   - description 第13行把「给我说说XX」列为触发词（暗示直接执行）
   - 两处矛盾：测试员无法判断「给我说说XX」是否需要追问

2. **追问顺序不够优化**
   - 场景1：「这事你怎么看？」连主题都没有，追问"决策还是学习"和"决策场景"意义不大
   - 应该先问「你说的'这事'是指什么？」，再进入标准追问流程

3. **混合意图没有处理规则**
   - 场景4暴露了规则盲区：「顺便看看YY」是多意图还是多事件蒸馏？没有判断逻辑

4. **反向蒸馏的触发词不够明确**
   - 「有没有类似的案例可以参考」已经暗示了反向蒸馏，但 Skill 可能从其他维度切入

### 改进建议

1. **统一模糊输入的定义**
   - 修改 description 第13行：「给我说说XX」→ 「给我说说XX」（如果用户说明了想学什么，才触发；没说意图时走追问流程）
   - 或者：把「给我说说XX」改为「帮我深入了解XX的决策逻辑」，明确意图

2. **优化追问流程**
   - 增加**前置判断**：输入是否有明确主题？
     - 无主题 → 先问「你说的'这事'是指？」
     - 有主题但无意图 → 标准3问
     - 有主题有意图 → 直接进入对应蒸馏模式

3. **增加混合意图处理规则**
   - 当用户提到多个事件时，询问：「你提到的这两件事，是同一类决策问题的不同案例，还是两个独立的问题？」
   - 如果是同一决策问题 → 多事件蒸馏
   - 如果是独立问题 → 分别处理或询问优先级

4. **明确反向蒸馏的优先级**
   - 当用户说「有没有类似案例」时，优先识别为反向蒸馏
   - 其他决策维度（竞争博弈、战略选择等）作为反向蒸馏的补充维度
