# 太史公 Skill 问题分析与改进建议

**发现时间**：2026-04-22
**发现问题**：商业用户-A 测试中，单事件蒸馏得出「控制权执念是致命弱点」的片面结论

---

## 一、问题清单

### 问题1：单事件蒸馏的样本量陷阱

**现象**：
- Skill 在单事件蒸馏模式下，得出「创始人对控制权的执念会成为致命弱点」的结论
- 但刘强东同样有控制权执念，京东却活了下来

**根因**：
- 质量控制文档（quality-control.md）写了「防幸存者偏差」，但**没有规定执行动作**
- 「应找反例」只是原则性描述，没有强制要求
- 规律验证只标注「弱—样本量不足」，但**没有引导用户补充样本**

**影响**：
- 单事件蒸馏的结论可靠性低，但用户可能不知道如何补充
- 信任度评级「低」的信息价值有限，用户决策参考意义不足

**改进建议**：

```
在 references/distillation-sop.md 中增加：

### 单事件蒸馏的强制动作
当执行单事件蒸馏时，必须执行以下步骤之一：

1. **主动匹配反例**（适用于涉及「成功/失败」判断的规律）
   - 如果提炼的规律暗示「某种做法导致失败」，必须主动搜索「同样做法但成功」的案例
   - 如果提炼的规律暗示「某种做法导致成功」，必须主动搜索「同样做法但失败」的案例
   
2. **标注关键差异维度**
   - 在输出中明确说明：「这个规律成立的前提是 [X]，如果 [Y] 则可能失效」
   - 给出判断方法：用户如何判断自己的情况属于 X 还是 Y

3. **给出补充样本建议**
   - 当样本量不足时，明确告知用户：「建议对比 [X] 案例，以验证此规律在你情况下的适用性」
   - 提供对比维度清单

示例问题维度：
- 控制权执念 → 需要对比：AB股 vs 否决权的区别
- 创始人决策 → 需要对比：被收购 vs 独立发展的不同结局
- 资本博弈 → 需要对比：多方博弈 vs 单一资本
```

### 问题2：规律描述的语言陷阱

**现象**：
- 原规律：「创始人对控制权的执念，在多方资本博弈格局下会成为致命弱点」
- 这个描述暗示「控制权执念 = 负面因素」

**根因**：
- 规律描述带有隐含的价值判断（执念是负面的）
- 没有区分「控制权的实现方式」和「要不要控制权」这两个层次

**改进建议**：

```
在 references/quality-control.md 中增加：

### 规律描述的中立性原则
1. **区分机制和结果**：描述机制（AB股 vs 否决权）而不是价值（执念）
2. **避免隐含因果**：用「在 [条件] 下，[机制] 导致 [结果]」而不是「[行为] 导致 [结果]」
3. **给出结构性变量**：列出影响结果的关键变量，而不是单一的因果判断

反例（错误）：
- 「创始人的控制欲是致命弱点」

正例（正确）：
- 「多票否决权在多方战略资本博弈格局下容易导致公司僵局」
- 「AB股结构与单层股权结构相比，在资本博弈中的效果差异取决于 [X] 变量」
```

---

### 问题3：反向蒸馏的主动关联缺失

**现象**：
- 用户描述了处境：「有大公司想收购我们」
- Skill 执行了单事件蒸馏，但没有主动将用户的处境与蒸馏结论关联

**根因**：
- output-template.md 中有「反向蒸馏输出格式」，但只在「反向蒸馏模式」下使用
- 当用户指定了具体事件时，Skill 按「经验学习」处理，没有额外关联用户处境

**改进建议**：

```
在 SKILL.md 工作流中增加：

### 混合意图的处理规则
当用户同时满足以下条件时，执行「单事件蒸馏 + 处境关联」：
1. 用户指定了具体事件（如 Ofo 崩溃）
2. 用户描述了自己的处境（如「我面临收购选择」）

处理方式：
1. 执行标准单事件蒸馏
2. 在输出末尾增加「你的处境 vs 本案例」对比段：
   - 相似维度：[用户处境与案例的共同点]
   - 差异维度：[用户处境与案例的不同点，以及这意味着什么]
   - 参考建议：「基于本案例，以下 [X] 条规律可能对你有参考价值」

示例格式：
> **你的处境关联**
> 你提到面临收购邀约，与本案例的相似点：[X]
> 差异点：[Y] — 这意味着 [Z]
> 建议参考的规律：[规律1]、[规律2]，因为 [理由]
```

---

### 问题4：信任度评级的信息价值不足

**现象**：
- 单事件蒸馏的信任度是「低—样本量不足」
- 但这个信息对用户来说只是一个标签，不知道该怎么处理

**改进建议**：

```
在 references/quality-control.md 中增加：

### 信任度评级后的行动建议
信任度评级不应只是结论，应包含对用户的行动指引：

| 评级 | 用户应该如何使用 |
|------|----------------|
| 高 | 可作为重要参考，但需结合自身情况判断边界 |
| 中 | 规律方向可信，但具体数值/程度可能不适用 |
| 低 | 仅作为思考框架，建议主动寻找对比案例验证 |

对于「低」评级，必须提供：
1. 「建议对比的案例类型」清单
2. 「建议核实的关键变量」清单
3. 「可以反驳此规律的潜在反例」描述
```

### 2.2 工作流补充

在「阶段三：决策拆解与蒸馏」中增加：

```markdown
### 强制样本检查（新增）

在提炼规律之前，执行以下检查：

1. **反例搜索**：对于涉及「成功/失败」「好/坏」「应该/不应该」判断的规律，
   必须用以下方式验证：
   - 如果规律暗示「X 导致失败」，搜索「X 但成功」的案例
   - 如果规律暗示「X 导致成功」，搜索「X 但失败」的案例
   
2. **关键变量识别**：识别导致不同结果的关键变量，而不是简单地说「X 导致 Y」
   
3. **适用边界标注**：对于每个规律，明确标注：
   - 在什么条件下成立
   - 在什么条件下可能失效
   - 用户如何判断自己属于哪种情况
```

## 三、references 文件的修改建议

### 3.1 distillation-sop.md 补充

```markdown
### 反例搜索指令

对于每个核心规律，执行以下搜索之一：

**如果规律是「X 导致正面结果」**：
- 搜索：「X 导致失败案例」
- 分析：失败案例中的关键差异变量

**如果规律是「X 导致负面结果」**：
- 搜索：「X 导致成功案例」
- 分析：成功案例中的关键差异变量

**如果规律是「X 和 Y 相关」**：
- 搜索：「X 和 Y 不相关的案例」
- 分析：什么条件下两者脱钩

搜索后，更新规律的「适用/不适用」描述。
```

### 3.2 quality-control.md 补充

```markdown
### 质控四要素的执行检查清单

**1. 防事后诸葛亮**
- [ ] 是否标注了「决策者当时知道 X，不知道 Y」？
- [ ] 反事实推演是否明确标注「这是推测，不是事实」？

**2. 防幸存者偏差**
- [ ] 是否主动搜索了「同样做法但不同结果」的案例？
- [ ] 规律是否包含了「条件 A 时成立，条件 B 时失效」？

**3. 防信息源偏差**
- [ ] 是否覆盖了至少 3 个不同立场的来源？
- [ ] 不同来源的冲突信息是否被标注？

**4. 防过度抽象**
- [ ] 每条规律是否能回溯到具体的事件细节？
- [ ] 规律的适用边界是否明确且可操作？

**对于单事件蒸馏，增加以下检查**：
- [ ] 是否有「样本量不足」的标注？
- [ ] 是否提供了「建议对比的案例类型」？
- [ ] 规律描述是否避免了「执念」「弱点」等带有价值判断的词汇？
```

## 五、执行验证

完成以上修改后，应执行以下测试验证：

1. **反例搜索测试**：用「控制权」类问题测试，验证是否主动搜索了反例
2. **规律中立性测试**：检查输出的规律描述是否避免了价值判断词汇
3. **处境关联测试**：用「我面临收购选择 + 指定案例」格式测试，验证是否自动关联
4. **信任度行动测试**：检查低信任度评级是否包含行动建议
