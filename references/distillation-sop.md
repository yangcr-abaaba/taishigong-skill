# 事件蒸馏SOP

## 六步蒸馏流程

### 第1步：事件结构化分析

**这一步的目的是建立事件的分析骨架，避免后续的决策拆解变成散点式的归纳。**

#### 1.1 事件背景梳理（所有事件必做）

在深入分析之前，先还原事件发生的「初始条件」：

| 维度 | 需要回答的问题 |
|------|--------------|
| 时间背景 | 事件发生在什么时期？当时的大环境是什么？（经济周期、技术阶段、政策周期等） |
| 领域背景 | 事件发生在哪个领域/市场/地区？该领域的竞争格局、成熟度、规则如何？ |
| 参与者格局 | 主要参与方有哪些？各自的资源、地位、诉求是什么？力量对比如何？ |
| 信息环境 | 当时的主流认知是什么？有哪些信息是公开的，哪些是内幕？信息不对称程度如何？ |

#### 1.2 选择分析框架

分析框架的选择取决于两个因素：**事件本身的性质**（事件类型）和**用户关心的维度**（决策维度）。事件性质决定哪种框架能拆解事件，决策维度决定在哪个框架下聚焦什么。

不同事件适用不同的分析框架，**选择最贴合事件本质的框架，不强制套用所有框架**：

**框架A：矛盾分析法**
- 适用场景：利益冲突明确的事件（商业竞争、政治博弈、军事对抗、资源争夺）
- 核心问题：什么是驱动事态发展的关键冲突？哪一方在主导？主导方是否发生了转换？
- 产出：主要矛盾/次要矛盾、矛盾的主要方面/次要方面、矛盾演化阶段
- 例：Ofo崩溃 → 主要矛盾"规模扩张 vs 现金流约束"，次要矛盾包括团队控制权之争（从属于主要矛盾）

**框架B：决策链分析法**
- 适用场景：决策失误或决策成功驱动结果的事件（技术选型错误、战略转型、工程事故）
- 核心问题：哪个决策改变了事件的走向？决策者当时的信息状态如何？如果选另一个方案会怎样？
- 产出：关键决策节点序列、信息不完备还原、因果链（A决策导致B后果）、真正的转折点
- 例：挑战者号灾难 → 不适合用矛盾分析，更适合分析决策链（O型环的知情与放行、组织中的信息传递断裂）

**框架C：因果链分析法**
- 适用场景：多重因素共同作用导致结果的事件（金融危机、公共卫生事件、系统性失败）
- 核心问题：哪些因素是充分条件？哪些是必要条件？因素之间的因果关系还是相关关系？
- 产出：因果链图、必要因素/充分因素区分、因果关系的方向和强度
- 例：2008年金融危机 → 不适合单一矛盾分析，需要分析次贷→证券化→评级失灵→杠杆崩塌的多因素因果链

**框架D：利益相关方分析法**
- 适用场景：多方博弈、政策制定、公共事件
- 核心问题：每个参与方的核心利益是什么？谁有否决权？谁的诉求被忽略了？联盟关系如何？
- 产出：利益相关方图谱（利益、影响力、立场）、联盟关系、权力动态
- 例：GDPR制定 → 多方利益博弈（欧盟委员会/科技企业/隐私倡导组织/各成员国），不适合用单一矛盾框架

**框架选择指南**：
- **事件性质优先**：先根据事件类型选最适配的框架（上表），再根据决策维度微调
- **决策维度微调**：同一种事件，不同决策维度可能偏好不同框架。如商业事件 + 竞争博弈维度偏矛盾分析，但商业事件 + 危机应对维度可能偏决策链分析（聚焦"出了事之后做了什么决策"）
- **不适配信号**：如果选定框架后分析进展困难、结论流于表面，说明框架可能选错了，应切换

**组合使用**：
- 复杂事件可以使用多个框架组合。如"诺基亚衰败"可以用框架A（功能机 vs 智能手机的矛盾）+ 框架B（拒绝触屏的决策链）+ 框架D（运营商、开发者、供应链的利益格局）
- 但不要为了"全面"而堆框架——选1-2个最贴合的即可

#### 1.3 识别关键转折（所有框架通用的输出要求）

无论选择哪个框架，第1步的最终产出必须包含：

1. **事件的核心驱动力是什么？**（一句话概括，不是"发生了什么"，而是"什么驱动了事态发展"）
2. **事件经历了哪几个阶段？**（每个阶段的关键变化是什么）
3. **真正的转折点在哪里？**（从这一刻开始，结果是否已基本注定？）

**这三个问题的答案就是后续「决策拆解」的路线图**——决策节点应与分析框架识别出的转折点对应。

### 第2步：信息采集

**2.1 搜索策略**
根据事件类型执行差异化搜索，详见 `event-type-guide.md`。

核心搜索角度（每个事件至少覆盖3个）：
- **事件经过**：时间线、关键节点、涉及角色
- **决策内幕**：当事人视角、决策会议记录、内部信息
- **第三方分析**：行业分析师、学术研究、深度调查报道
- **多方对比**：支持性观点 vs 批判性观点 vs 中立分析

**2.2 信息筛选标准**
| 优先级 | 来源类型 | 示例 |
|--------|---------|------|
| P0 | 官方报告/调查报告 | 事故调查报告、财报、政策文件 |
| P1 | 权威媒体深度报道 | 财新、纽约时报深度、Wired长文 |
| P2 | 当事人一手叙述 | 传记、自传、播客访谈、公开演讲 |
| P3 | 分析师/学术研究 | 券商报告、学术论文、行业白皮书 |
| P4 | 综合性/二手来源 | 维基百科、教科书、综述文章 |

**2.3 信息记录**
搜索过程中记录：
- 每条关键信息的来源URL或出处
- 来源的立场倾向（当事方/支持方/反对方/中立）
- 信息的信任度（多方验证/单一来源/存疑）

**搜索的局限**：能搜到的信息取决于搜索引擎的覆盖范围、语言偏好和时效性。非英语事件、小众事件、新近事件的公开资料可能不足。搜索结果本身可能有误，交叉验证能降低但不能消除信息偏差。如果关键信息缺失，在输出中如实标注，不要用推测填补信息空白。

### 第3步：决策拆解

**基于第1步的结构化分析，决策节点应与分析框架识别出的关键转折点对应。**

**3.1 识别关键决策节点**
不是列出所有决策，只提取对结果有**重大影响**的3-5个决策节点。

判断标准：
- 去掉这个决策，结果是否会显著不同？
- 这个决策是否代表了某种战略选择或价值观取舍？
- 这个决策是否对应第1步分析框架中的某个关键转折？

**3.2 每个节点的结构化分析**

| 字段 | 说明 |
|------|------|
| 决策时间 | 精确到月，标注距关键结果的时间距离 |
| 决策者 | 具体到人/团队/组织 |
| 决策内容 | 具体做了什么选择 |
| 当时信息 | 决策者当时知道什么？不知道什么？（关键：还原信息不完备状态） |
| 面临压力 | 时间压力、资源约束、利益相关方博弈、情感因素 |
| 可选方案 | 当时有哪些备选项？（至少列出2个） |
| 选择理由 | 为什么选了这个而不是其他方案 |
| 实际结果 | 这个决策最终导致了什么 |
| 预期 vs 实际 | 结果是否符合预期？偏差在哪里？ |

**3.3 构建决策链**
将多个决策节点按时间串联，识别：
- 哪些决策是因果关联的（A决策导致必须做B决策）
- 哪些决策是独立失误（即使其他决策都对，这个也会导致失败）
- 哪个决策是真正的转折点（从这一刻开始，结果已基本注定）

### 第4步：反事实推演

**4.1 针对每个关键决策节点，推演替代方案**
- 如果选了另一个方案，最可能的结果是什么？
- 推演时必须基于当时的信息条件，不能用上帝视角
- **硬性约束**：替代方案必须是当时决策者真正考虑过的选项之一，而非后世旁观者脑补的方案。如果历史记录无法确认决策者是否考虑过某方案，必须标注「推演方案」

**4.2 整体层面的反事实**
- 如果改变一个关键决策，整个事件的走向会怎样？
- 有没有「无论怎么选都会失败」的死局？（如果有，说明问题出在更早的阶段）

**4.3 反事实的验证**
- 类似情境下，有没有人做过替代选择？结果如何？
- 如果没有历史先例，标注为「未经验证的推演」

**反事实推演的本质局限**：反事实推演是基于当时条件的推测，不是可靠预测。替代方案的真实结果永远无法知道。推演的价值在于暴露「当时决策者面临的真正权衡」，而不是预测「另一条路一定通向什么」。

### 第5步：规律提炼

**5.1 从具体到抽象**
- 从决策拆解中提取可复用的模式
- 每条规律必须满足：能脱离这个具体事件独立存在

**5.1.1 单事件蒸馏强制反例搜索**

单事件蒸馏只有一个样本，任何结论都可能是片面的。因此，提炼出规律后**必须**执行反例搜索：

1. 对每条提炼出的规律，搜索是否存在反例：「有没有在类似条件下做了相同决策但结果不同的事件？」
2. 如果找到反例：标注该规律的局限性，降级为条件性规律（需补充「在X条件下成立，在Y条件下可能失效」）
3. 如果找不到反例：标注「未找到反例验证」（注意：「未找到」不等于「不存在」，仅说明搜索范围内未见反例）
4. 如果规律被反例推翻：放弃该规律，或将其收窄到更精确的条件范围

**执行标准**：反例搜索至少尝试3次不同关键词组合。如果3次搜索均未找到反例，可标注「当前搜索未发现反例」，但不应因此将弱规律升级为强规律。

**5.2 规律的结构化表述**

每条规律包含：
```
规律描述：[一句话概括]
来源：[来自哪个事件的哪个决策节点]
适用条件：[在什么情况下这个规律成立]
失效条件：[在什么情况下这个规律不适用]
```

多事件蒸馏中，额外标注：
```
验证事件：[还有哪些事件验证或推翻了这个规律]
```

**5.3 规律的分级**
| 级别 | 标准 | 示例 |
|------|------|------|
| 强规律 | 被3个以上不同事件验证 | 「规模不是护城河，现金流才是」|
| 中规律 | 被2个事件验证，或被1个强事件+理论支撑 | 「给更多比拿走的阻力小」|
| 弱规律 | 仅被1个事件验证，无反例但也无更多支持 | 需标注「样本量不足，谨慎参考」|

如果提炼不出有价值的规律，如实告知。不要为了输出而强行抽象。

**规律的本质局限**：从历史事件提炼的规律是概率性的模式，不是必然成立的定律。适用条件标注得再完整，也无法穷举所有边界情况。单事件蒸馏只能产出弱规律（样本量不足），多事件蒸馏能提高规律可靠性，但仍不等于因果定律。

### 第6步：结构化输出

按 `output-template.md` 格式输出决策参考卡片。

输出前最终检查：
- [ ] 输出能独立使用吗？（不依赖用户看过原始资料）
- [ ] 每条规律有适用边界吗？
- [ ] 有信任度评级吗？
- [ ] 决策者当时的信息不完备状态还原了吗？
- [ ] 有没有事后诸葛亮的语言？（如「显然应该...」「很明显...」）
- [ ] 事件背景梳理完整吗？（时间、领域、参与者、信息环境至少覆盖3个维度）
- [ ] 选择了合适的分析框架吗？（不是机械套用，而是匹配事件类型）
- [ ] 决策节点与分析框架识别出的转折点对应了吗？（不是孤立的决策列表）

> 以上检查项与五要素质控的映射关系：
> - 「信息不完备状态还原」→ 要素一：防事后诸葛亮
> - 「事后诸葛亮语言」→ 要素一：防事后诸葛亮
> - 「规律有适用边界」「每条规律能回溯到具体事件」→ 要素四：防过度抽象
> - 「来源多样性」「多方验证」→ 要素三：防信息源偏差
> - 「反例搜索」「规律验证」→ 要素二：防幸存者偏差

## 多事件蒸馏流程

当用户提出决策场景但未指定具体事件时，在六步蒸馏流程基础上执行多事件蒸馏：

### M1. 决策场景解析

从用户输入中提取决策场景的核心特征：
- 用户面临什么类型的决策？（竞争博弈/战略选择/危机应对/政策影响）
- 决策的关键约束是什么？（资源、时间、信息不完备程度）
- 用户最关心哪个维度？（风险控制/收益最大化/多方平衡/长期影响）

### M2. 事件匹配

遵循 SKILL.md 中的多事件匹配逻辑：
1. 优先在同类型事件池中匹配最相关的案例
2. 跨类型仅用于补充同类型事件缺乏的视角
3. 事件数量由决策场景决定，不设硬性约束
4. 主动搜索不同结果的事件（成功+失败），避免偏差

**匹配时的注意事项**：
- 如果所有匹配事件都指向同一结论，主动搜索反例事件
- 记录每个匹配事件的匹配理由（为什么选这个事件）
- 标注每个事件的信息完整度（高/中/低），让用户知道哪些事件的分析基础更扎实

### M3. 逐个蒸馏

对每个匹配事件执行单事件蒸馏（第1-5步）。可以使用简要版以控制时间，但以下内容不可省略：
- 事件结构化分析（至少选框架+识别转折点）
- 关键决策拆解（至少2个核心决策节点）
- 核心规律提炼（每条带适用边界）
- 单事件信任度评级

### M4. 交叉验证

对比多个事件的规律，执行以下分析：

1. **共识规律**：被多个事件验证的规律
   - 标注：哪些事件支持，支持程度如何
   - 共识规律的可信度最高，但需检查是否存在「所有来源都偏向同一叙事」的偏差

2. **矛盾发现**：不同事件指向不同结论的地方
   - 必须解释矛盾原因（通常是约束条件不同：资源、时机、市场环境、政策背景等）
   - 给出用户视角的判断指引（在什么条件下参考哪个结论）

3. **条件边界**：规律在什么条件下成立/失效
   - 从矛盾发现中抽象出关键变量
   - 标注用户可以通过哪些信号判断自己属于哪个条件

### M5. 多事件综合输出

按 `output-template.md` 中多事件模板格式输出。输出前额外检查：
- [ ] 事件概览表中每个事件都有信息完整度标注
- [ ] 共识规律有验证事件列表
- [ ] 矛盾发现有矛盾原因和用户判断指引
- [ ] 有多事件专用信任度评级（维度：事件覆盖度、跨类型验证、规律可靠性、信息来源）

## 反向蒸馏流程

当用户描述自己的处境时，执行反向蒸馏：

### R1. 用户处境结构化

从用户描述中提取：
- 行业/领域
- 发展阶段（早期/成长期/成熟期/衰退期）
- 可用资源（资金、人才、时间、技术）
- 核心困境（矛盾的主要方面）
- 决策约束（不可改变的硬约束 vs 可以调整的软约束）

如果用户描述信息不足，追问以下关键信息：
- 行业/领域是什么？
- 公司/组织规模？
- 当前最紧迫的压力是什么？
- 有哪些不可改变的约束条件？

### R2. 历史案例匹配

1. 基于用户处境的关键特征，在5类事件池中搜索匹配事件
2. **匹配数量**：至少3个。如果少于3个，进入降级处理（见R5）
3. **匹配质量要求**：
   - 每个匹配事件的核心困境与用户处境有实质性相似（不是表面相似）
   - 主动搜索不同结果的事件（有人成功应对了类似困境，有人失败了）
   - 记录每个案例的匹配理由

### R3. 案例蒸馏与对比

对每个匹配案例：
1. 简要蒸馏（决策概况 + 核心教训）
2. 对比分析：
   - **相似点**：用户的处境与历史案例的共同点
   - **差异点**：不同之处 → **必须标注对用户的影响**（差异意味着什么，可能导致什么不同的结果）
3. 从每个案例中提取1-2条与用户处境直接相关的启示

### R4. 综合启示

基于多个案例的交叉对比，输出：
1. 多个案例一致指向的启示（可信度最高）
2. 案例间存在矛盾的启示（标注矛盾条件和用户判断指引）
3. 用户特有的风险点（来自差异点分析）

**免责声明**：必须明确标注「以上是基于历史案例的启示，不是对你的决策建议。你的具体环境与历史案例存在差异，最终决策需要结合自身情况判断。」

### R5. 降级处理

当以下情况发生时，触发降级：
- 搜索匹配到的案例少于3个
- 匹配案例与用户处境的相关度低
- 匹配案例的信息公开资料严重不足

**降级处理方式**：
1. 告知用户：「当前场景的历史匹配案例不足，以下分析基于有限的案例，可靠性较低。」
2. 输出已找到的案例（即使不完整，也比没有好）
3. 建议用户：
   - 提供更具体的行业/决策背景以提高匹配精度
   - 直接指定想要了解的事件（切换为单事件蒸馏）
   - 寻找行业内部的专业咨询（公开资料可能不足的领域）

### R6. 反向蒸馏输出

按 `output-template.md` 中反向蒸馏模板格式输出。输出前额外检查：
- [ ] 每个匹配案例都有相似点和差异点分析
- [ ] 差异点标注了对用户的影响
- [ ] 有综合启示
- [ ] 有免责声明
- [ ] 有信任度评级（维度：匹配相关度、信息充分度、案例多样性、规律可靠性）

## 辅助分析流程

当用户要求分析个人事件时（信息获取难、可靠性低），降级为辅助分析模式：

### 辅助分析的适用条件
- 用户提供的事件属于个人/私密事件
- 公开资料极少或不存在
- 用户自己提供了详细背景信息

### 辅助分析流程
1. 告知用户：「这是一个个人事件，公开可查的资料极少，以下分析基于你提供的信息和有限的公开资料，可靠性低于公众事件分析。」
2. 如果用户提供了背景信息，基于用户提供的信息进行决策逻辑拆解（跳过信息采集步骤，改为标注「信息来源：用户自述」）
3. 按六步蒸馏流程执行，但以下调整：
   - 跳过搜索策略，信息来源标注为「用户自述 + 有限公开资料」
   - 信任度评级默认不超过「中」
   - 每条规律额外标注「基于个人事件，样本量不足，仅供参考」
4. 如果用户未提供足够背景信息，追问关键信息后再分析

### 辅助分析的输出格式
使用单事件蒸馏模板，但增加以下标注：
- 在信任度评级前增加：「⚠️ 本分析基于个人事件，可靠性低于公众事件分析。」
- 信任度评级的「信息来源」维度默认为「低」，其他维度根据实际分析质量评定
